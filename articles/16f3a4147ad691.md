## キャッシュロジック実装の刷新

Pythonのキャッシュライブラリ `beautyspot` の v2.7.6 をリリースした。

今回のアップデートでは、目に見える新機能の追加ではなく、非同期処理やデータベース制御といった「裏側」の安定性を固めることに注力している。

`beautyspot` v2.7.6 では、関数の実行結果を裏側で非同期に保存する。

しかし、この「裏側でよしなにやる」という挙動は、プロセスの終了時やマルチスレッド環境下において、データの不整合やクラッシュを招く要因にもなり得る。

本記事では、v2.7.6 で実施した内部設計の刷新と、それを支える理論的な仕組みについて、主要な3つのポイントに絞って整理する。

## 1. SQLite接続の「安全なクローズ」とスレッドセーフティ

`beautyspot` の `SQLiteTaskDB` は、並行読み取り性能を高めるためにスレッドごとに接続を保持する設計をとっている。しかし、シャットダウン時にこれらの接続を一括で閉じようとすると、以下の技術的課題に直面した。

*   **スレッドの制約**: SQLite の接続は、作成したスレッド以外から `close()` を呼ぶとエラー（またはセグメンテーションフォールト）を誘発するリスクがある。
*   **デッドロック**: エラーリカバリ中に再帰的にロックを取得しようとしてハングアップする。

### 対処：ノンブロッキング・クローズ機構
v2.7.6 では、リエントラントロック（`RLock`）と**ノンブロッキング・クローズ**を組み合わせた。

シャットダウンや GC（ガベージコレクション）時のクローズ処理では、ロック取得を `blocking=False` で試行する。もし他のスレッドがクエリを実行中でロックが取れない場合は、その場でのクローズを「安全に諦め」、Python の自然な GC 管理に委ねる設計とした（ADR-0049）。「無理に閉じようとして壊す」よりも「安全なタイミングまで遅延させる」という判断である。

## 2. 強参照セットによるタスクの「生存保証」

バックグラウンドで実行中の保存タスク（`Future`）をどのように追跡するか、という点も再設計の対象となった。

当初、メモリリークを恐れて `WeakSet`（弱参照）での管理を検討したが、これには「ユーザーコード側で Future の参照が切れた瞬間、完了を待機すべきタスクが GC によって勝手に追跡対象から消えてしまう」という非決定的なリスクが伴う。

### 対処：決定論的なライフサイクル管理
v2.7.6 では、通常の `set`（強参照）を使用し、タスク完了時のコールバックによって明示的にセットから削除する方式を採用した（ADR-0047）。

*   **生存保証**: `Spot` インスタンスが強参照を保持することで、タスクが確実に完了するまで GC されないことを物理的に保証する。
*   **待機可能性**: `flush()` 呼び出し時に、未完了のタスクを漏れなく検知し、確実にドレイン（排出）できる。

これにより、タスクのライフサイクルが GC という外部要因から切り離され、挙動の予測可能性が向上した。

## 3. シャットダウン・シーケンスの厳密な定義

`beautyspot` は内部で「イベントループ（`_BackgroundLoop`）」と「スレッドプール（`Executor`）」の2つを組み合わせて動いている。プロセスの終了時にこれらをどの順序で止めるかは、リソースのクリーンアップにおいて極めて重要である。

### 対処：Loop -> Executor の順序維持と防御的コード
もし先に `Executor` を強制終了させてしまうと、ループ側で実行中の I/O タスクが `CancelledError` や `RuntimeError` を吐き出し、一時ファイルの削除漏れなどのノイズが発生する。

v2.7.6 では、以下のシャットダウン順序を厳密に実装した（ADR-0051）。
1.  **Loop の停止**: 新規タスクの受け入れを停止し、現在のキューをドレインする。
2.  **Executor の停止**: I/O 待ちのタスクを安全に終了させる。

同時に、`run_in_executor` を呼び出す箇所に防御的な `try...except` ブロックを配置し、強制終了時であってもログを汚さず、優雅に（Graceful）システムを停止させる仕組みを整えた。

## まとめ

*   **SQLite管理**: 強制クローズによるクラッシュを避け、ノンブロッキングな試行と GC への委譲を選択。
*   **タスク追跡**: 弱参照の不確実性を捨て、強参照とコールバックによる決定論的な管理へ移行。
*   **シャットダウン**: ループとエグゼキュータの依存関係に基づいた適切な停止シーケンスを確立。

`beautyspot` が目指すのは、開発者が「キャッシュの挙動」を気にする必要さえない、空気のような存在である。v2.7.6 は、その理想に近づくための、静かだが重要な一歩となった。
