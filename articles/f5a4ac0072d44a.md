# Pythonにおける「決定論的」なキャッシュキー生成：引数の正規化とバージョニングの罠

## キャッシュキー生成は、見た目ほど簡単ではない。

関数の引数からキャッシュキーを作る。一見、`hash(args)` を取るだけの単純な作業に思えるが、実務でこれを実装しようとすると、無数の「非決定的な要素」に足元を掬われることになる。

Python の標準的な `hash()` 関数は、セキュリティ上の理由（Hash Flooding 攻撃対策）から、実行プロセスごとにシード値が変わる。つまり、プロセスを再起動した瞬間に全てのキャッシュキーが無効化されてしまうのだ。

本記事では、ライブラリ `beautyspot` において、どのような工夫で「永続的かつ衝突の少ない」キャッシュキー生成を実現したかを整理する。

## 1. `inspect.signature` による引数の正規化

キャッシュにおいて最大の敵は、「実質的に同じ呼び出しなのに、キーが異なる」ことだ。

```python
def func(a, b=10): pass

# これらはすべて同じ結果を返すべきだが、素直に引数を取るとキーが分かれる
func(1, 10)
func(1)
func(a=1, b=10)
func(b=10, a=1)
```

これを解決するために、Python の `inspect.signature` を使用して引数を「正規化」するプロセスを導入した。

```python
import inspect

sig = inspect.signature(func)
bound_args = sig.bind(*args, **kwargs)
bound_args.apply_defaults()
# ここで bound_args.arguments は常に {'a': 1, 'b': 10} という順序付き辞書になる
```

呼び出し方がどうであれ、インターフェース定義に基づいた唯一の「正典（Canonical Form）」に変換することで、キーワード引数の順序やデフォルト値の有無によるキーの不一致を完全に排除した（ADR-0010）。

## 2. 複雑なオブジェクトの安定ハッシュ化

次に直面するのが、Pydantic モデルや Dataclass、あるいは巨大な辞書といった「ハッシュ化可能（Hashable）ではない」オブジェクトの扱いだ。

`beautyspot` では、これらのオブジェクトを再帰的に走査し、決定論的なシリアライズを行う仕組みを構築した。

*   **辞書のソート**: 辞書は必ずキーでソートして処理する。
*   **型情報の付与**: `1`（整数）と `"1"`（文字列）が同じハッシュにならないよう、型情報をシリアライズに含める。
*   **Msgpack の活用**: 高速かつコンパクトなバイナリ形式である Msgpack をベースに、特定の型（Pydantic等）を扱うエンコーダを組み合わせることで、安定したバイト列を生成し、その SHA-256 を取る設計とした。

これにより、メモリ上のオブジェクトの状態が「論理的に等価」であれば、常に同じキャッシュキーが得られるようになった。

## 3. 「ロジックの変更」をどう検知するか

キャッシュの寿命において最も難しいのが、関数の中身（ロジック）を書き換えた時だ。

引数が同じでも、計算ロジックが変われば、古いキャッシュを返してはならない。しかし、Python において「関数のバイトコードが変わったこと」を自動で完璧に検知するのは非常にコストが高く、かつ副作用（クロージャの参照先の変化など）を追いきれない。

`beautyspot` では、あえて**明示的なバージョニング**という「規約」を選択した。

```python
@spot.mark(version="v1")
def my_logic(data):
    ...
```

ユーザーが `version` を変えることで、キャッシュキーのプレフィックスが切り替わる。自動検知という「魔法」を捨て、明示的な管理という「予測可能性」を取る。この判断が、大規模なパイプラインにおいて「なぜか古い結果が返ってくる」というデバッグ困難なバグを防ぐ防波堤となっている（ADR-0003）。

## 4. ストレージパスとキーの衝突回避

生成されたハッシュ（SHA-256）は、最終的にデータベースの主キーや、ファイルシステムのパスとして使用される。

ここで重要なのは、衝突（Collision）の可能性をゼロと見なさないことだ。非常に低い確率ではあるが、ハッシュが一致しても内容が異なる可能性に備え、DB にはハッシュだけでなく、元となった「関数名」や「入力の識別子」をメタデータとして保持させている。これにより、万が一の衝突時にもデータの混同を防ぐためのガードレールを設けている。

## まとめ

決定論的なキャッシュキーを生成するためには、Python の動的な性質を「正規化」というプロセスで押さえ込む必要がある。

*   **`inspect`** で引数の呼び出し方を統一する。
*   **ソート済みのシリアライズ**でオブジェクトの状態を固定する。
*   **明示的なバージョン管理**でロジックの変更を制御する。

これらの積み重ねが、数万件のキャッシュを抱えても揺るがない、ライブラリの信頼性を支えている。
