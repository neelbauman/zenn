# SQLiteを高性能キャッシュエンジンとして飼い慣らす：並行処理と書き込み遅延の克服

## SQLiteは「簡単」だが「高速」ではない、という誤解。

Python でキャッシュライブラリ `beautyspot` を開発するにあたり、バックエンドのストレージとして SQLite を採用した。理由は単純で、追加のミドルウェア（Redis等）を必要とせず、ファイル一つで完結し、かつ SQL による強力なクエリが使えるからだ。

しかし、デフォルト設定の SQLite をマルチスレッド環境下でそのままキャッシュエンジンとして使おうとすると、すぐに「Database is locked」というエラーに直面することになる。

本記事では、SQLite を「低レイテンシかつ並行処理に強いキャッシュ基盤」へと変貌させるために施した、設計上の工夫を整理する。

## 1. WALモードの採用とチェックポイントの罠

並行性能を確保するための第一歩は、ジャーナリングモードを `WAL (Write-Ahead Logging)` に変更することだ。

```sql
PRAGMA journal_mode = WAL;
PRAGMA synchronous = NORMAL;
```

WAL モードを有効にすると、読み取りと書き込みが互いをブロックしなくなる。キャッシュエンジンにおいて、データの保存（書き込み）を行っている間も検索（読み取り）が継続できることは、性能上の必須要件である。

ただし、WAL モードには「チェックポイント」という管理コストが伴う。書き込みデータはまず `-wal` ファイルに蓄積され、適切なタイミングで本体の `.db` ファイルへ書き戻される。この書き戻しが正常に行われないと、`-wal` ファイルは肥大化し続け、逆に読み取り性能を悪化させる。すべての接続を適切にクローズしなければチェックポイントが完了しないという特性は、マルチスレッド環境での接続管理に重い課題を突きつけてくる。

## 2. スレッドローカル接続と「クローズ」のジレンマ

SQLite の接続（Connection）は、原則として作成したスレッド以外で共有してはならない。`check_same_thread=False` というオプションはあるものの、これを安易に使うとクラッシュの原因となる。

`beautyspot` では、`threading.local()` を用いてスレッドごとに専用の接続を保持する設計をとった。

```python
import threading
import sqlite3

class SQLiteTaskDB:
    def __init__(self, path):
        self._local = threading.local()
        self._path = path

    def _get_conn(self):
        if not hasattr(self._local, "conn"):
            self._local.conn = sqlite3.connect(self._path)
        return self._local.conn
```

ここで問題になるのが、スレッドが終了する際の接続の破棄だ。接続が残り続けると前述の WAL チェックポイントが走らない。しかし、シャットダウン時に他スレッドの接続を無理に閉じようとすれば、`sqlite3.ProgrammingError` やセグメンテーションフォールトを引き起こす。

この問題に対し、v2.7.6 では「ノンブロッキング・クローズ」を選択した（ADR-0049）。ロックが取れない（＝他スレッドが使用中）なら、その場でのクローズを諦め、Python の GC に処理を委ねる。この「無理をしない」設計が、結果としてシステムの堅牢性を高めることになった。

## 3. 専用 Writer スレッドによる「書き込みの直列化」

SQLite において、最大のボトルネックは「書き込み」である。複数のスレッドから同時に `INSERT` や `UPDATE` を試みると、SQLite のロック機構により待ち時間が発生し、アプリケーションのレスポンスを悪化させる。

これを解決するために導入したのが、**「専用 Writer スレッド」** というパターンだ（ADR-0041）。

メインスレッドや作業スレッドは、自ら `execute("INSERT...")` を実行しない。代わりに、保存したいデータをスレッドセーフな `queue.Queue` に投げ入れ、即座に本来の処理に戻る。

```python
def _writer_loop(self):
    conn = sqlite3.connect(self._path)
    while True:
        task = self._queue.get()
        if task is None: break
        # ここで唯一の書き込みスレッドとしてSQLを実行
        conn.execute(...)
        conn.commit()
```

この設計により、以下のメリットが得られた。

*   **メインスレッドのレイテンシゼロ**: DBの書き込み完了を待がずに次の処理へ進める。
*   **書き込み競合の完全排除**: 書き込みを単一のスレッドに集約することで、SQLite レベルでのロック競合が発生しなくなる。
*   **一括処理（バッチング）の余地**: 負荷が高い場合に、複数のタスクを一つのトランザクションでコミットする最適化が可能になる。

## 4. 安定性を担保する「防御的プログラミング」

最後に、キャッシュエンジンとして運用する上で不可欠なのが、異常系への対応だ。

キャッシュはあくまで「補助的なデータ」であるべきで、DB の不調が原因でメインのアプリケーションがクラッシュしては本末転倒である。`beautyspot` では、DB 操作の失敗を捉え、それをバックグラウンドで通知（`on_background_error`）しつつ、メイン処理には「キャッシュミス」として透過的に復帰させる仕組みを整えた。

## まとめ

SQLite は、適切に飼い慣らせば、ローカル環境において Redis にも引けを取らない強力なキャッシュエンジンになる。

*   **WALモード**で並行性を確保する。
*   **Writerスレッド**を分離し、書き込みを直列化してメインスレッドを解放する。
*   **接続管理**は「安全な諦め」を含めた防御的な設計にする。

これらの設計判断の積み重ねが、`beautyspot` の「空気のような使い心地」を支えている。
