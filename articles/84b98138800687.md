# 確率的エビクション：メインスレッドのレイテンシを1msも犠牲にしないストレージ管理

## キャッシュの掃除は、いつやるべきか。

キャッシュライブラリ `beautyspot` を運用していると、避けて通れないのが「ストレージの肥大化」だ。数万件の LLM レスポンスをキャッシュすれば、当然ながらディスクやデータベースの容量を圧迫し始める。

しかし、キャッシュを保存するたびに「期限切れのデータはないか？」「容量は上限を超えていないか？」とチェックし、不要なファイルを削除するのは非常に重い処理だ。キャッシュライブラリの使命は「実行を速くすること」なのに、そのメンテナンスのためにメインの関数のレイテンシが悪化しては本末転倒である。

本記事では、このジレンマを解決するために採用した **確率的エビクション（Probabilistic Auto-Eviction）** の設計思想と実装について整理する。

## 1. 毎回のチェックが引き起こす「マイクロ遅延」の蓄積

最も素朴な実装は、`save()` を呼ぶたびにクリーンアップロジックを走らせることだ。

一回の削除処理が 10ms だとしても、それが毎回積み重なれば、スループットは確実に低下する。特に LLM アプリケーションのように、数秒かかる処理の後に「たかが 10ms」と思うかもしれないが、その 10ms の積み重ねがシステム全体の「キレ」を奪っていく。

かといって、全く掃除をしなければストレージは無限に膨らむ。定期的なバッチ処理（Cron等）に頼る手もあるが、ライブラリ単体で「自己完結」したメンテナンス能力を持たせたい、という思いがあった。

## 2. 「1%の確率」という合理的妥協

そこで導入したのが、**「100回に1回だけ掃除する」** という確率的なアプローチだ（ADR-0036）。

```python
import random

def should_evict(rate: float = 0.01) -> bool:
    # 1% の確率で True を返す
    return random.random() < rate
```

この「1%」という数字には、明確な合理性がある。
*   **計算コストの分散**: 99% の呼び出しにおいて、クリーンアップのオーバーヘッドはほぼゼロになる。
*   **統計的な収束**: 呼び出し回数が増えるほど、クリーンアップの頻度は期待値（1%）に収束し、ストレージの肥大化を一定の範囲内に抑え込める。

「毎回 0.1ms かける」のではなく「100回に1回 10ms かける」。平均的なコストは同じだが、多くのリクエストにおいてレイテンシの「最良値」を維持できるのが、この手法の強みだ。

## 3. バックグラウンドへのオフロードと「ゾンビ・コレクション」

さらに `beautyspot` では、この確率的なチェックにあたった際も、クリーンアップ処理をメインスレッドで行わない。

専用のメンテナンスサービス（`MaintenanceService`）を介して、バックグラウンドのワーカースレッドへ処理を委譲する。
*   **非同期実行**: メインスレッドは「掃除を予約した」だけで即座に復帰する。
*   **再帰的削除**: データベースから削除されたが、ファイルシステムに残っている「ゾンビファイル」を、バックグラウンドで静かに回収していく。

この「確率的発火 + 非同期実行」の組み合わせにより、ユーザーはメンテナンスの存在を一切意識することなく、常にクリーンなストレージの状態を享受できるようになった。

## 4. ユーザーに委ねる「掃除の頻度」

もちろん、アプリケーションの特性によって最適な頻度は異なる。
*   **短期集中型**: 頻繁に保存が発生するため、`eviction_rate=0.001` (0.1%) 程度で十分。
*   **低頻度・大容量型**: 一回の保存が重いため、`eviction_rate=0.05` (5%) 程度で早めに掃除したい。

`Spot` クラスの初期化時にこのレートを調整可能にすることで、パフォーマンスとメンテナンス性のトレードオフをユーザーが制御できるようにしている。

## まとめ

ライブラリの運用において、メンテナンスは「不可欠だが邪魔なもの」だ。

*   **確率的エビクション** により、平均的なオーバーヘッドを極小化する。
*   **非同期処理** と組み合わせ、メインスレッドのレイテンシを完全に保護する。
*   **統計的な管理** によって、外部のバッチ処理に依存しない自己完結型のライブラリを実現する。

「見えないところで、静かに、確実に」。この哲学が、ストレージ管理においても貫かれている。
