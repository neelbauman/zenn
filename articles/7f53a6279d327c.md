# ライブラリのための「非侵入的」な依存性注入：利便性と柔軟性を両立する設計

## ライブラリにDIフレームワークを持ち込みたくなかった。

ライブラリ `beautyspot` を設計する際、避けたかったことの一つが、特定の DI（依存性注入）フレームワークをユーザーに強いることだ。キャッシュの保存先をローカルから S3 に変えたい、あるいは DB を SQLite から Redis に変えたいといった要求は当然発生する。しかし、そのためにユーザーに複雑な設定ファイルを強いたり、ライブラリのコアが巨大な依存関係を抱えたりするのは、私たちの「空気のような使い心地」という哲学に反する。

本記事では、外部フレームワークに頼らず、Python の標準的な機能を組み合わせて実現した「非侵入的」な DI アーキテクチャを整理する。

## 1. Protocol による「静的なダックタイピング」

まず、各コンポーネント（DB、ストレージ、シリアライザ）の境界線を `typing.Protocol` で定義した。

```python
from typing import Protocol

class TaskDBBase(Protocol):
    def get(self, cache_key: str) -> str | None: ...
    def save(self, **kwargs) -> None: ...
```

継承（`abc.ABC`）ではなく `Protocol` を採用した理由は、ユーザーが既存の自作クラスを `beautyspot` に持ち込みやすくするためだ。明示的にライブラリの基底クラスを継承していなくても、メソッドシグネチャさえ合っていれば DI 可能になる。この「疎結合」な関係が、ライブラリの侵入性を最小限に抑えている（ADR-0024）。

## 2. ファクトリ関数による「デフォルト DI」の隠蔽

DI は柔軟性を生むが、一方で「使う前の準備（ボイラープレート）」を増やしてしまう。ユーザーがいきなり S3 や Redis を使いたいわけではなく、まずは「とりあえず動かしたい」のが本音だ。

そこで、`Spot` クラスの初期化時にデフォルトのコンポーネントを自動で組み立てるファクトリ関数を導入した。

```python
class Spot:
    def __init__(self, name, db=None, storage=None):
        # 明示的に渡されなければ、デフォルトの構成をその場で生成する
        self.db = db or create_default_db(name)
        self.storage = storage or create_default_storage(name)
```

ユーザーは `Spot(name="app")` と書くだけで、背後で SQLite とローカルストレージが適切に注入された状態で使い始めることができる。内部的な複雑さをデフォルト値の裏に隠し、必要になった時だけ「注入」して上書きする。この「設定より規約（Convention over Configuration）」に近いアプローチが、導入のハードルを劇的に下げている（ADR-0025）。

## 3. 「卒業のしやすさ」とロックインの回避

ライブラリ設計において重要なのは、ユーザーがそのライブラリから「卒業」する時のことまで考えることだ。

もし `beautyspot` のコアが特定のクラウドベンダーの SDK に深く依存していたら、プロジェクトが成長して自前の大規模な基盤へ移行しようとした時、ライブラリそのものが足枷になってしまう。

コンポーネントを DI 可能な構造に保つことは、ユーザーがいつでも「特定のバックエンド」から離れられることを意味する。
*   **開発時**: ローカルストレージで高速に試行錯誤。
*   **本番運用**: 注入するインスタンスを S3 バックエンドに差し替えるだけ。
*   **卒業時**: デコレータを外し、注入していたロジックを自前のコードに組み込む。

この「脱着の容易さ」こそが、ライブラリがユーザーの信頼を勝ち取るための誠実な設計であると考えている。

## 4. 依存関係の「遅延インポート」戦略

DI をサポートするために、オプションの依存関係（例: `boto3`）をコアの `dependencies` に含めないようにした。

特定のバックエンドを使用しようとした瞬間に初めてインポートを試み、インストールされていなければ適切なエラーメッセージを表示する。これにより、デフォルトのインストールサイズを最小限に保ちつつ、必要に応じて「プラグイン的」に機能を拡張できる構造を実現した。

## まとめ

ライブラリにおける DI は、フレームワークを導入することではなく、コンポーネント間の「境界線」を正しく引くことである。

*   **Protocol** でインターフェースを疎結合に定義する。
*   **ファクトリ関数**でデフォルトの挙動を隠蔽し、UX を守る。
*   **遅延インポート**でコアの依存関係をクリーンに保つ。

「いつでも差し替えられる」という安心感が、結果としてユーザーがライブラリを長く、深く使い続けるための基盤となっている。
